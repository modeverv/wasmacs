# Minibuffer Command Loop Plan

## Goal

Move from the current browser echo line to real Emacs-owned minibuffer
behavior without letting the browser become a parallel minibuffer
implementation.

The current `#minibuffer` surface is only an echo/display slice. It may show
prefixes and explicit unavailable messages, but it must not own file-name
reading, buffer-name reading, completion, minibuffer history, recursive
minibuffer depth, or selected-window state.

## Source Grounding

- `vendor/emacs/lisp/files.el` routes `find-file` through `read-file-name`.
- `vendor/emacs/lisp/window.el` and `vendor/emacs/lisp/simple.el` route
  buffer selection through `read-buffer`, `completing-read`, and
  `read-from-minibuffer`.
- `vendor/emacs/lisp/minibuffer.el` defines completion behavior and minibuffer
  setup hooks around the Lisp reader surface.
- `vendor/emacs/src/minibuf.c` owns the C-side active minibuffer stack:
  `Vminibuffer_list`, `minibuf_level`, `active-minibuffer-window`,
  `minibufferp`, `read_minibuf`, and the selected minibuffer window.
- `vendor/emacs/src/keyboard.c` owns the command loop shape that minibuffer
  reads depend on.
- `vendor/emacs/src/window.c` owns selected-window and minibuffer-window state
  needed to display and restore the active minibuffer.

## Required Boundary

Real minibuffer support requires an Emacs-side command loop that can pause for
browser input and resume the same Emacs state. The browser host should expose
minibuffer state as a display/input protocol, not as a second implementation of
Emacs readers.

First protocol shape:

```text
host.gui.minibuffer-state
  active: boolean
  prompt: string
  input: string
  point: integer
  depth: integer
  completing: boolean
  completion_table_available: boolean
  unavailable_reason: string | nil

host.gui.minibuffer-input
  key event / text input / accept / quit
```

The browser may render this state and send input events back. It may not choose
file candidates, mutate history, or decide `find-file` / `switch-buffer`
results outside Emacs.

## Incremental Phases

1. Echo-only boundary:
   - keep `C-x C-f` and `C-x b` as explicit unavailable commands
   - show `C-x` prefix and unavailable messages in `#minibuffer`
   - verify with `npm run browser:smoke:all`
2. State reporting probe:
   - add an Emacs-side probe that can report inactive minibuffer state:
     `active-minibuffer-window`, `minibuffer-depth`, and current echo text
   - keep real `read-from-minibuffer` unavailable
3. Suspended read design:
   - design a host entrypoint that can enter `read_minibuf`, yield to browser
     input, and resume without losing C stack/GC root safety
   - this must be tied to command-loop-level state from `keyboard.c`
4. First real command:
   - enable `C-x C-f` only after `read-file-name` can be driven by Emacs state
   - browser file picker/list remains a convenience path, not minibuffer
     semantics
5. Completion and history:
   - expose completion candidates generated by Emacs
   - preserve minibuffer history variables on the Emacs side

## Current Blockers

- There is not yet a resumable host command loop for `read_minibuf`.
- Host eval still needs careful stack/GC root handling when long-lived
  Emacs-owned state crosses calls.
- Browser input can send key events, but the active selected window /
  minibuffer window relationship is not yet represented as a protocol state.

Until those are addressed, minibuffer-facing commands remain explicit
unavailable boundaries.
